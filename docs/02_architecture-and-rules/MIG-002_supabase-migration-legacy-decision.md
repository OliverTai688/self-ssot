# D-DECISION-002: Supabase Migration Legacy Strategy

**Task:** DB-005
**Date:** 2026-06-09
**Status:** DECIDED — no action required; legacy file retained as reference

---

## Context

`supabase/migrations/20260520000000_init.sql` was created before the Prisma-first migration strategy was adopted. It defines the initial schema using raw SQL (Supabase's native migration format). The Prisma baseline migration (`prisma/migrations/`) was generated separately from `prisma/schema.prisma`.

These two migration files represent the same intended schema but were generated by different tools and may have subtle differences (column ordering, index naming, extension syntax).

---

## Current State

| File | Tool | Status |
|---|---|---|
| `supabase/migrations/20260520000000_init.sql` | Supabase CLI raw SQL | Legacy / reference only |
| `prisma/migrations/` | Prisma Migrate | Canonical — use this |
| `prisma/schema.prisma` | Prisma ORM | Source of truth |

The Supabase migration file was NOT applied to any production database during the current development cycle. The Prisma baseline migration is the reviewed and approved path forward.

---

## Decision

**Retain the Supabase migration file as a reference artifact. Do not delete or apply it.**

Rationale:
- The file documents the original schema intent before Prisma adoption
- It may be useful if Supabase-specific features (RLS, functions, triggers) need to be added alongside Prisma migrations in the future
- Deleting it does not improve anything; retaining it costs nothing

**The Supabase migration file is NOT part of the active migration path.** It must not be applied via `supabase db push` without first reconciling it against the current Prisma schema and verifying that all differences are intentional.

---

## Identified Differences

The Supabase migration and the Prisma baseline may differ in:
- Extension loading syntax (`CREATE EXTENSION IF NOT EXISTS "uuid-ossp"` vs `pgcrypto`)
- Enum naming: Supabase SQL uses `snake_case` but Prisma generates uppercase enum values by default (see DB-004 decision)
- Index names: Prisma auto-generates index names; Supabase SQL may use explicit names
- No `_prisma_migrations` table in the Supabase file (expected — Prisma manages this separately)

---

## Required Before Any Supabase Apply

If Supabase-native features (RLS policies, edge functions, webhooks) are required:
1. Reconcile `20260520000000_init.sql` against the current Prisma schema
2. Split into: (a) Prisma-managed tables (do not duplicate), (b) Supabase-only extensions (RLS, policies, triggers)
3. Create a new `supabase/migrations/` file for part (b) only
4. Review with DBAgent before applying to any non-disposable database
