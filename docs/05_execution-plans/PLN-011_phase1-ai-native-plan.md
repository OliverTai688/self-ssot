## 1. 產品定位重設

### 一句話定位

> **Personal OS 是一個 AI 協作型個人作業系統，持續在背景理解你的工作、研究與生活脈絡，主動整理混亂資訊，幫你看見重要連結，讓你把認知資源留給真正需要你做決定的事。**

### 這不是什麼

| 容易被誤解成 | 真正應該是 |
|-------------|------------|
| 任務管理工具（待辦清單） | 脈絡整理助理（AI 整理，你決定）|
| CRM 聯絡人資料庫 | 關係流動觀察員（AI 看見誰和誰有連結）|
| 研究資料庫（論文、材料） | 研究地圖助理（AI 幫你判斷方向與成熟度）|
| 健康追蹤器（KPI 式提醒）| 生活照護管家（溫和照顧，不是監控）|
| 財務管理軟體（報表） | 營運節奏助理（AI 讀懂現金流訊號）|
| 筆記工具（手動分類） | 外部記憶容器（AI 結構化你的碎片輸入）|

### 根本轉移

**舊思維：** 使用者輸入資料 → 系統儲存 → 使用者操作介面取出

**新思維：** 使用者丟入碎片 → AI 理解脈絡 → AI 整理成結構 → 使用者確認或修正 → 系統行動

這個轉移意味著：

- UI 不是主角，AI 的判斷與整理才是主角
- 表格和 Dialog 是承載 AI 輸出的容器，不是使用者的主要工作介面
- 使用者的主要動作是**確認、修正、拒絕**，而不是**輸入、分類、填寫**

## 2. 核心 AI 體驗哲學

### 2.1 六個設計原則

**Principle 1：Capture cheap，Review easy**

捕捉的摩擦力要趨近於零。一段 LINE 訊息、一個語音備忘、一張收據照片、一個散步中浮現的研究想法——這些都應該能在 10 秒內進入系統，AI 負責後續的理解與整理。使用者的主要動作是用 3 秒確認 AI 的判斷，不是用 3 分鐘填寫表單。

**Principle 2：Narrative over Numbers**

Dashboard 上的數字卡片是舊思維。AI 應該給你的是**敘事**：「你的 Lisa Dashboard 專案本週有新訊息但三個任務卡住了，可能需要你確認一個客戶問題。」數字是數字，但脈絡是理解。

**Principle 3：Confirmation over Creation**

使用者的角色是**審閱者與決策者**，不是資料輸入員。AI 提案，使用者確認。這是 Human-in-the-loop 的正確位置——不是讓使用者一直操作 AI，而是讓 AI 一直為使用者準備好草稿、分類、建議，使用者只在判斷點進場。

**Principle 4：Context over Categories**

傳統系統的分類是靜態標籤（工作 / 研究 / 生活）。但現實是：一個想法可能同時是研究材料、也是工作提案的基礎、也和某個商會關係有關。AI 應該看見這些**橫向連結**，不是強迫你選一個資料夾。

**Principle 5：Proactive over Reactive**

好的個人 OS 不等你來問。它在你開機時已經整理好「今天你應該知道什麼」。它在一段時間後說「你的研究主軸 A 已經三週沒有更新，而你最近丟進來的三個想法跟它有關」。它主動浮現，不是等你搜尋。

**Principle 6：Gentle over Urgent**

特別是生活管理與健康追蹤。提醒不能是壓力，不能是未讀數量焦慮，不能是 KPI 感。AI 的語氣應該像一個關心你的人，而不是一個效率顧問。

### 2.2 AI 如何降低認知負荷

認知負荷來自三個地方：

1. **切換負荷**：從研究者切換到 CEO 再切換到商會理事，每次切換都要重新建立脈絡
2. **決策負荷**：每件事都要自己判斷輕重緩急
3. **追蹤負荷**：很多事情「不記得在哪裡」「不確定還在不在進行」

AI 的角色不是幫你「做事」，而是幫你「知道現在在哪裡、接下來要往哪」。它讓你每次打開系統時，不需要重新建立脈絡——因為 AI 已經幫你保持著脈絡。

## 3. AI 如何重新定義「每日開機」

### 3.1 舊模式 vs 新模式

**舊模式（Phase 1 原計劃的 Dashboard）：**
```
4 個統計卡：進行中專案數 / 本週到期任務 / Inbox 數 / 生活提醒
→ 使用者自己從數字判斷今天要做什麼
```

這等於把認知負荷丟還給使用者。

**新模式：AI Morning Brief**
```
AI 說：
「今天是 5 月 5 日，早上好。

這週有三件事在你的注意圈內移動：

[工作] Lisa 的 Q2 Dashboard 專案——她昨天傳了一則 LINE 訊息，
       你上週回應後還有一個她的問題尚未確認。
       AI 已整理好回應建議草稿，需要你過目。

[研究] 你上週語音備忘了一個關於「社群媒體中介效果」的想法，
       AI 發現它和你的研究主軸 B 的核心問題有重疊，
       同時找到 2 篇相關論文，已準備好讓你決定要不要納入。

[生活] 你上次看醫生是三週前，你當時說要追蹤恢復狀況。
       今天可能是個合適的時機想一想。

其他事項安靜地在追蹤中，今天不需要特別處理。
想看完整的今日脈絡地圖嗎？」
```

### 3.2 Morning Brief 的結構

Morning Brief 不是清單，是**敘事性摘要**，由以下元素組成：

| 元素 | AI 做什麼 | 使用者做什麼 |
|------|----------|-------------|
| **今日注意圈** | 從所有模組中挑出 2–4 個真正需要人類注意的事 | 決定要先處理哪一個 |
| **有變化的脈絡** | 標示哪些專案/關係/研究有新訊號 | 選擇要深入看 |
| **AI 已備好的草稿** | 回應建議、分類建議、摘要 | 確認或修改 |
| **靜默追蹤確認** | 告知哪些事情 AI 在追蹤中，今天不需要你管 | 知道就好，不需操作 |

### 3.3 重要設計細節

- **有邊界感**：Morning Brief 應該有個結束感，像 OpenAI Pulse 的設計——它告訴你「今天就這些」，讓你知道已經掌握全局，而不是讓你感覺永遠看不完
- **不強迫行動**：每個 Brief 條目都有「之後再說」的選項，不製造焦慮
- **解釋 AI 為什麼說這個**：每個提醒都要說明觸發原因（「因為 Lisa 的訊息」「因為你上次說要追蹤」），這是信任感的基礎

## 4. AI Triage：重新設計 Inbox

### 4.1 舊 Inbox vs AI Triage Inbox

**舊設計（Phase 1 原計劃）：**
- 使用者看到一串 Inbox 條目
- 手動從下拉選單選「歸入哪個模組」
- 等同於讓使用者自己分類

**新設計：AI Intake → AI Triage → Human Review**

使用者丟進任何碎片，AI 在背景完成三層處理：

```
[Layer 1 - Intake] 原始捕捉
  使用者丟入：LINE 訊息 / 語音備忘 / 照片 / URL / 隨手文字

[Layer 2 - AI Triage] AI 處理（幾秒內）
  ├── 判斷類型：任務 / 專案紀錄 / 研究想法 / 關係互動 / 財務 / 生活 / 回憶素材
  ├── 提取實體：人名 / 專案名稱 / 金額 / 日期 / 關鍵詞
  ├── 判斷緊急度：需要今天處理 / 本週 / 無期限
  ├── 建議歸屬：哪個專案 / 哪個研究主軸 / 哪個聯絡人
  ├── 生成摘要：一句話說明這筆資料的意義
  └── 建議下一步：AI 認為接下來應該做什麼

[Layer 3 - Human Review] 使用者審閱
  每筆顯示 AI 的判斷結果，使用者只需三個動作：
  ├── ✅ 確認（全部同意 AI 的建議）
  ├── ✏️ 修正（調整分類或下一步）
  └── ❌ 跳過（暫時不處理）
```

### 4.2 AI Triage 頁面體驗設計

每筆 Triage 條目不是一列資料，而是一張**AI 整理卡片**：

```
┌─────────────────────────────────────────────┐
│ [LINE 圖示] 來自客戶群組 · 剛才             │
│                                             │
│ 原始訊息：                                  │
│ "這個 banner 我覺得太大了" + 一張截圖       │
│                                             │
│ ── AI 整理 ──────────────────────────────── │
│ 類型：[客戶意見回饋]  緊急度：[本週內]      │
│ 歸入：[Lisa Q2 Dashboard 專案] > [紀錄]     │
│ 提取：Lisa・banner 尺寸問題・待確認         │
│                                             │
│ AI 建議下一步：                             │
│ 「回覆 Lisa 確認 banner 規格，或請設計師    │
│   先提供三個尺寸選項。已準備好回應草稿。」  │
│                                             │
│ [✅ 確認並歸入]  [✏️ 修改]  [查看草稿]      │
└─────────────────────────────────────────────┘
```

### 4.3 Triage 的 AI 判斷依據（需要透明）

每個 AI 判斷旁邊要有一個「為什麼」的展開：

```
AI 為什麼這樣分類？
→ 「因為這則訊息來自客戶群組，你最近在工作模組中
    有一個名稱包含 'Lisa' 的進行中專案。訊息中有
    產品名稱 'banner'，判斷為客戶意見回饋類型。」
```

這是信任感的來源。使用者知道 AI 怎麼判斷的，才會放心讓它整理。

### 4.4 Batch Triage 流程

對於早上一次性審閱多筆 Inbox，設計類似「快速審閱模式」：

- 一次顯示 3–5 筆，使用者快速滑動確認
- 有「全部接受 AI 建議」的選項（高信心的條目）
- 低信心的條目（AI 說「不確定」）自動排到後面等使用者親自處理

### 4.5 Phase 1 模擬方式

- AI 判斷結果以**預寫的 mock 回應**呈現，體現 AI 分析的 UI 模式
- 不同類型的輸入（任務、想法、訊息、金額）有不同的 AI 卡片格式
- 使用者確認/修改的互動邏輯完整可運作（前端 state）
- 「查看草稿」打開一個 AI 預寫的建議文字（mock）

## 5. AI Project Companion：工作專案體驗

### 5.1 概念轉移

舊：專案 = 任務列表 + 紀錄列表 + 交付物列表（三個分開的 Tab）

新：專案 = 一個**有生命的脈絡空間**，AI 持續在其中幫你理解狀態

### 5.2 AI Project Companion 提供什麼

進入一個專案時，AI 先說話：

```
[AI 專案摘要 · 今天]

Lisa Q2 Dashboard 專案目前狀態：

✅ 進展中：前端框架已完成，設計稿確認中
⚠️ 待確認：Lisa 昨天提出 banner 尺寸問題，尚未回覆
⛔ 阻塞中：後端 API 等待客戶提供資料格式文件（已等待 5 天）

AI 建議今天優先處理：
1. 回覆 Lisa 的 banner 問題（已準備草稿）
2. 向 Lisa 確認 API 文件時程

距離交付日還有 12 天，目前進度：約 60%
```

這個摘要從哪裡來？
- 從所有 `project_notes`（含 LINE 訊息來源）中提取
- 從 `project_tasks` 的狀態分析
- 從最後更新時間判斷哪些事情「停滯了」

### 5.3 內部視角 vs 客戶視角的 AI 轉化

這是整個工作模組最有價值的 AI 能力：

**內部視角**（只有你看得到）：
```
Lisa 這個月脾氣比較急，可能是她的主管在施壓。
banner 問題可能只是表面，背後可能是整體設計方向的不確定。
建議先打電話溝通，再用 LINE 確認文字。
```

**AI 客戶版本**（AI 自動幫你轉化）：
```
本週專案進度：
- 前端框架完成 ✅
- 正在處理您提到的 banner 尺寸，我們準備了三個選項讓您確認
- API 整合預計下週開始，需要您提供一份資料格式文件

預計下次提供成果：5/10
```

使用者不需要自己重寫，只需要按「確認送出」或微調措辭。

### 5.4 AI Project Companion 的具體 UI 表現

- 專案頁面頂部：**AI Pulse**（AI 狀態摘要，3–5 行，每天更新）
- 下方才是原來的任務、紀錄、交付物——但這些是**原始資料層**，使用者想看的時候才看，不是主要視角
- **「AI 幫我寫客戶更新」按鈕**：一鍵生成草稿，供確認後使用
- **「哪裡卡住了？」**：AI 分析所有超過 X 天沒有更新的任務，提出阻塞點

### 5.5 Phase 1 模擬方式

- AI Pulse 是預寫的 mock 文字，每個 mock 專案各有一份
- 「AI 幫我寫客戶更新」開啟一個 Dialog 顯示 mock 草稿
- 使用者可以在 Dialog 中編輯文字（前端可運作的 textarea）
- 任務列表、紀錄等保留，但不是主要入口，是「查看原始資料」的選項

## 6. AI Research Cartographer：研究地圖體驗

### 6.1 概念轉移

舊：研究主軸 = 想法列表 + 材料列表 + 研究者列表（資料庫）

新：研究主軸 = **知識地圖**，AI 幫你看見想法之間的連結、判斷成熟度、建議路徑

### 6.2 AI Research Cartographer 提供什麼

**① 想法歸屬判斷**

當使用者丟進一個想法：「社群媒體的演算法設計是否影響集體行動的可能性？」

AI 說：
```
這個想法與以下研究主軸有關：
→ 主軸 A「平台中介與社會資本」（高度相關）
→ 主軸 C「集體行動的數位轉化」（中度相關）

同時，這個想法有跨領域轉化潛力：
→ 可以作為工作上與 NGO 客戶提案的理論基礎（見 Lisa NGO 專案）
→ 適合發展成 LinkedIn 文章（科技與社會議題，你的常見發表形式）

建議歸入主軸 A，並在主軸 C 標記為「關聯想法」。
```

**② 研究成熟度評估**

每個研究主軸，AI 根據累積的想法、材料、里程碑進度，給出一個成熟度判斷：

```
主軸 A「平台中介與社會資本」

成熟度：★★★☆☆（概念成形期）
AI 判斷：你有 12 個相關想法、5 篇材料，但假設尚未明確，
         缺乏實證設計。目前比較適合繼續探索，而非開始寫作。

下一步建議：
→ 明確核心研究問題（你的想法目前散布在 3 個不同問法）
→ 找一篇能代表當前研究前沿的論文，做為定錨
→ AI 發現你的想法和 Chen et al. 2024 的研究框架高度重疊，要看看嗎？
```

**③ 論文 → 概念連結**

當使用者貼入一篇論文 URL：

```
AI 解析：
標題：「Platform Governance and Collective Action」
作者：Chen, Wang, Liu · 2024 · ACM CHI
摘要摘要：探討平台設計如何影響使用者集體行動能力...

與你的研究：
→ 強化主軸 A 的理論框架
→ 作者 Chen 與你的研究者資料庫中的 David Wang 有合著關係
→ 這篇論文提出的「演算法中介」概念，與你上週的語音備忘高度呼應

要將此論文加入主軸 A 的材料，並建立和 David Wang 的關聯嗎？
```

### 6.3 Phase 1 模擬方式

- 每個研究主軸有一份 mock 的 AI 成熟度評估文字
- 「丟入想法」的 Triage 流程中，AI 卡片會顯示「建議歸入哪個主軸，以及為什麼」（mock）
- 論文 URL 輸入後，顯示一個 mock 的 AI 解析卡片
- 研究者連結（「Chen 和你的 David Wang 有合著關係」）以 mock 資料體現

## 7. AI Relationship Sensemaker：商會與關係體驗

### 7.1 概念轉移

舊：聯絡人 = 姓名 + 公司 + Email + 電話（通訊錄）

新：關係 = **動態的人際脈絡**，AI 看見關係的流動、引薦的適切時機、合作的可能性

### 7.2 AI Relationship Sensemaker 提供什麼

**① 關係動態觀察**

AI 定期（每週）整理：

```
本週關係動態觀察：

升溫：
→ Allen（科技業）最近在商會群分享了三篇相關文章，
  可能對你的工作更感興趣了。建議主動聯繫。

冷卻：
→ David（顧問）已超過 45 天沒有互動。
  你們上次見面有提到一個合作想法，可能值得跟進。

引薦時機：
→ Lisa（設計師）和 Mark（品牌顧問）分別提到過類似的需求，
  AI 認為可能有合作機會，你想了解更多嗎？
```

**② 引薦 DM 生成**

不是選範本填空，而是 AI 理解脈絡後生成：

```
你想把 Allen 引薦給 Lisa。

AI 生成引薦 DM：
「Lisa，想介紹一位朋友給你認識——Allen 在科技領域耕耘多年，
 最近他們公司正在找設計夥伴協助產品體驗優化，
 我想到你的專長可能很對應，所以引薦一下！
 Allen 的 LINE 是 @allen.xxx，你們可以直接約聊。」

這個版本的根據：
→ Lisa 是設計師，Allen 有設計需求
→ 你上次和 Allen 見面記錄中，他提到「找設計夥伴」

要直接複製，還是想調整語氣？
```

**③ 關係洞察（私人層）**

AI 從你的私人備注、互動紀錄、LINE 訊息中整理：

```
關於 Mark 的私人脈絡（僅你可見）：

他目前公司內部有人事變動，這讓他對外部合作更開放。
你們上次見面他表達過對你公司主軸的興趣，但沒有具體推進。
AI 建議：這段關係目前停在「暖場」階段，
         一個半正式的午餐邀請可能比業務 DM 更適合。
```

### 7.3 Phase 1 模擬方式

- 聯絡人列表保留，但**首頁入口改成「AI 本週關係觀察」**而非聯絡人總覽
- 每個聯絡人詳情頁頂部有 AI 的「關係摘要與建議」（mock）
- 引薦 DM 生成器：輸入兩個人名後，顯示 AI 生成的 mock DM 草稿
- 商機看板簡化為「AI 建議的合作機會」優先呈現，不是完整 Kanban

## 8. AI Operating Rhythm：公司與財務體驗

### 8.1 概念轉移

舊：公司管理 = 文件列表；財務 = 收支表格 + 圖表

新：公司與財務是**一個整體的營運節奏感**，AI 從中看見信號

### 8.2 每週公司節奏摘要

AI 每週一生成：

```
本週公司狀態（5/5 · 週一）

核心節奏：
→ 你這個月有 2 個進行中客戶專案，目前都在正常推進中
→ 新案子正在探索中（Allen 的引薦方向），尚未進入簽約

財務信號：
→ 本月收入目前為 $0（主要客戶款項在月底），
  這是正常節奏，但建議確認 Lisa 專案的付款日期
→ 有 2 筆支出尚未分類（5/2 的聚餐 $850、
  5/4 的雲端服務費 $299），需要你確認是內帳還是外帳

需要你決定的事：
→ 本月公司節奏是否正常？（你自己的感受最重要）
```

### 8.3 財務的 AI 角色：信號而非報表

財務模組的 AI 不是要幫你記帳，而是幫你：

1. **捕捉到記錄**：收據照片 → AI OCR → 自動建立草稿 → 你只需確認金額和類別
2. **從紀錄中看見信號**：哪個月支出異常、哪個專案的收款延遲了、現金流是否健康
3. **連結到專案**：這筆費用和哪個專案有關（AI 猜測，你確認）

不需要讓使用者主動打開財務頁面看圖表。財務信號應該出現在 Morning Brief 和公司節奏摘要中。

### 8.4 Phase 1 模擬方式

- 公司管理和財務**合併成一個「營運節奏」頁面**，頂部是 AI 週摘要（mock）
- 財務紀錄列表保留，但入口降格（不是主要視角）
- 「未分類支出」的 AI 提醒以橫幅形式出現在頁面頂部
- 圖表降級為可收合的次要區塊，不是主視覺

## 9. AI Life Keeper：生活照護與回憶體驗

### 9.1 概念轉移

舊：生活事項 = 看板 + 待辦 + KPI 式提醒

新：**照護而非追蹤**，AI 溫和地關心你有沒有照顧好自己

### 9.2 AI Life Keeper 的語氣設計

AI 在生活模組的語氣，要刻意和工作/研究模組不同：

**工作模組 AI 語氣：** 分析性、行動導向
```
「這個任務已延遲 5 天，建議今天先跟 Lisa 確認一下。」
```

**生活模組 AI 語氣：** 溫和、關心、非強迫
```
「你上次提到感冒快好了，是這樣嗎？
 如果還沒完全恢復，最近天氣變化大，可以注意一下。
 不用特別做什麼，只是想到你說的就問一下。」
```

### 9.3 生活照護的 AI 運作方式

- **不用使用者主動追蹤**：使用者在任何捕捉管道提到生活事項（「最近感冒」「修書包」「好久沒運動了」），AI 自動記錄並在適當時間溫和浮現
- **間隔提醒而非固定提醒**：不是「每週三提醒運動」，而是「你已經 14 天沒提到運動了，最近有機會嗎？」
- **不算分不排名**：沒有完成率、沒有連續幾天的 streak，因為這會產生焦慮

### 9.4 回憶管理的 AI 角色

**回憶捕捉：**

使用者丟入一張和朋友的照片或一段 LINE 對話截圖：

```
AI 說：
「這張照片是你和 Mark 在 5/1 勞動節的聚餐。
 你已經有 3 個和 Mark 相關的回憶，
 要把這張加入 Mark 的回憶集，還是單獨保存？

 如果你想，我可以幫你寫一段當時的記憶摘要，
 留住一些你現在還記得的細節。」
```

**回憶整理（禮物 / 紀念）：**

```
Mark 的生日快到了（5/20，還有 15 天）。

AI 根據你們過去的互動和共同回憶，整理了一些參考：
→ 你們聊過他很喜歡手沖咖啡
→ 今年 2 月你們一起去的展覽，你說他很感興趣
→ 你有一張你們 2024 年底的合照，還沒整理進回憶錄

不需要現在決定任何事。這只是讓你想到他的時候，
有個地方可以看看過去的記憶。
```

### 9.5 Phase 1 模擬方式

- 生活模組**頁面頂部改成 AI 的溫和問候**（mock），不是看板統計
- 生活事項列表簡化，類別看板保留但降格
- 回憶牆保留，但新增「AI 為某人整理的回憶摘要」作為展示 mock
- 生日/紀念日提醒：以 AI 關心型語氣出現，不是事件通知

## 10. 資料模型重設：AI 中心的流動單位

### 10.1 舊模型的問題

傳統 Project / Task / Contact / FinanceRecord 等表設計的問題：

- 它們是**靜態的存儲容器**，不描述資料的「流動」
- 它們無法表達**AI 的理解和建議**
- 它們不描述**使用者在哪裡做了決定**

### 10.2 AI 中心的九個資料單位

```
┌─────────────────────────────────────────────────────────┐
│  [Capture] → [AI Triage] → [Entity / Context Build]    │
│       ↓                           ↓                     │
│  [Insight]      ←────────────  [AI 分析]               │
│       ↓                                                  │
│  [Suggestion]   ←──── AI 提案                           │
│       ↓                                                  │
│  [Decision]     ←──── 使用者確認 / 修正 / 拒絕          │
│       ↓                                                  │
│  [Action]       ─────→ 執行（任務 / 記錄 / 草稿）       │
│  [Memory]       ─────→ 長期保存                         │
│  [PublicOutput] ─────→ 對外分享 / 交付                  │
└─────────────────────────────────────────────────────────┘
```

### 10.3 各單位定義

| 單位 | 定義 | 誰建立 | 誰使用 |
|------|------|--------|--------|
| **Capture** | 原始輸入，未加工的碎片 | 使用者（任何管道）| AI 讀取 |
| **Context** | AI 對某個 Entity 的當前理解狀態 | AI 建立與更新 | AI 推論、使用者查看 |
| **Entity** | 系統內的核心物件（人/專案/研究/生活事項/公司/財務）| AI 識別，使用者確認 | 所有操作的核心 |
| **Insight** | AI 從資料中提煉的洞察（關聯、趨勢、風險）| AI | 使用者決策輸入 |
| **Suggestion** | AI 對下一步行動的建議（含草稿）| AI | 使用者審閱與確認 |
| **Decision** | 使用者對 Suggestion 的回應（確認/修改/拒絕）| 使用者 | 訓練 AI 理解偏好 |
| **Action** | 被確認後的具體行動（任務、發送、記錄）| 使用者確認後系統執行 | 追蹤用 |
| **Memory** | 被選擇保存的重要片段（回憶、里程碑、洞察）| AI 建議，使用者確認 | 長期回顧 |
| **PublicOutput** | 可對外分享的內容（客戶更新、引薦 DM、研究摘要）| AI 草稿，使用者確認 | 對外溝通 |

### 10.4 資料流示意

一個 LINE 訊息如何流過這個系統：

```
1. [Capture] Lisa 在群組說「這個 banner 太大了」+ 截圖
      ↓
2. [AI Triage] 類型：客戶意見；提取：Lisa、banner、尺寸問題
      ↓
3. [Entity] 歸入「Lisa Q2 Dashboard 專案」這個 Entity
      ↓
4. [Context 更新] Lisa 專案的 Context 新增：「客戶對 banner 尺寸有疑慮」
      ↓
5. [Insight] AI：「這是本週第二次 Lisa 提到 UI 細節問題，
              可能有更深層的設計方向不確定性」
      ↓
6. [Suggestion] AI 準備兩個草稿：
              a. 客戶回覆 DM（提供三個尺寸選項）
              b. 內部任務（請設計師調整 banner）
      ↓
7. [Decision] 使用者確認草稿 a，修改草稿 b
      ↓
8. [Action] 任務建立、客戶 DM 草稿存入待發送
      ↓
9. [PublicOutput] 客戶更新摘要中加入：「正在處理 banner 尺寸問題」
```

傳統模式下，使用者需要：手動建任務、手動寫回覆、手動更新專案紀錄。
AI 模式下，使用者只需要：看 AI 的整理，確認或修改。

## 11. Phase 1 最小可感知 AI 體驗

### 11.1 Phase 1 的核心目標

> 不是「完成所有頁面」，而是「讓使用者感受到這個系統和他們用過的任何工具都不一樣」。

這個「不一樣」的感覺，只需要 3–5 個 AI 體驗時刻就夠了。

### 11.2 五個核心 AI 體驗情境

#### 情境 1：Morning Brief（每日 AI 摘要）

**使用者旅程：**
```
打開系統 → 看到 AI 的今日敘事摘要
→ 每條摘要可以「深入」（進入對應模組）或「之後再說」
→ 看完後有結束感（「今天的概覽到此，有需要再告訴我」）
```

**AI 輸入：** 所有 Entity 的 Context + 最近 Capture + 日期資訊

**AI 輸出：** 2–4 條敘事性摘要，各有一個「深入」和「之後再說」的操作

**Human-in-the-loop：** 每條可以「之後再說」，不強迫現在處理

**Phase 1 模擬方式：** 預寫的 mock Morning Brief，根據當天日期動態選擇（mock 資料有 5 組，週一/週三/週五/突發事件/輕鬆日）

---

#### 情境 2：AI Triage（捕捉 → AI 整理 → 確認）

**使用者旅程：**
```
Cmd+K 打開快速捕捉 → 輸入任何碎片（文字/URL）→ 送出
→ Inbox 出現一張 AI Triage 卡片
→ 卡片顯示：AI 的分類、提取的實體、建議歸入哪裡、建議下一步
→ 使用者：✅ 確認 / ✏️ 修改 / ❌ 跳過
```

**AI 輸入：** 原始文字 + 現有 Entity 列表（用來匹配歸屬）

**AI 輸出：** 分類 Badge + 實體提取 + 歸屬建議 + 下一步建議 + 說明「為什麼」

**Human-in-the-loop：** 每張卡片使用者都要確認，高信心條目可批量確認

**Phase 1 模擬方式：**
- 輸入框送出後，依輸入內容的關鍵詞觸發不同 mock AI 卡片（5–8 種模式）
- 確認後更新 Inbox 的 mock 狀態
- 「為什麼這樣分類」有展開說明（mock 但合理）

---

#### 情境 3：AI Project Pulse（專案 AI 狀態摘要）

**使用者旅程：**
```
進入任一專案 → 頂部看到 AI Pulse 摘要
→ 摘要說明：目前狀態、進展中、待確認、阻塞點、建議今天優先什麼
→ 點擊「AI 幫我寫客戶更新」→ Dialog 顯示 AI 草稿 → 編輯 → 複製
```

**AI 輸入：** 專案的所有 Notes + Tasks 狀態 + 最後更新時間

**AI 輸出：** 3–5 條狀態摘要 + 優先行動建議 + 客戶更新草稿

**Human-in-the-loop：** 客戶更新草稿在送出前必須人工確認並可編輯

**Phase 1 模擬方式：**
- 每個 mock 專案有預寫的 AI Pulse（不同狀態：健康/有阻塞/快到期）
- 客戶更新草稿是 mock 文字，可在 Dialog 中直接編輯（前端 textarea）
- 複製按鈕功能正常

---

#### 情境 4：AI Relationship Insight（引薦 DM 生成）

**使用者旅程：**
```
在商會頁面看到「AI 本週關係觀察」
→ AI 說：「Lisa 和 Mark 可能有合作機會」→ 點擊查看
→ 進入引薦 DM 生成：AI 自動填入兩人資料和脈絡
→ 一鍵複製 DM 文字
```

**AI 輸入：** 兩個 Contact 的資料 + 互動紀錄 + DM template

**AI 輸出：** 引薦 DM 草稿（個人化，非套版）

**Human-in-the-loop：** 使用者複製前可以在 textarea 修改

**Phase 1 模擬方式：** mock 關係建議 + mock DM 草稿，選擇兩個聯絡人後顯示

---

#### 情境 5：AI Memory Capture（AI 協助記憶保存）

**使用者旅程：**
```
從 Cmd+K 或 Triage 丟入一張照片描述或 LINE 對話片段
→ AI 說：「這看起來是你和 Mark 的聚餐，要我幫你寫一段記憶摘要嗎？」
→ AI 生成一段記憶描述
→ 使用者確認後加入回憶牆
```

**AI 輸入：** 使用者輸入的文字 + 已知的 Contact 資料

**AI 輸出：** 溫和的記憶摘要文字（情境性，不是資料性）

**Human-in-the-loop：** 使用者可以編輯 AI 的記憶描述，也可以直接拒絕

**Phase 1 模擬方式：** 輸入包含人名或朋友關鍵詞時，觸發 mock 記憶摘要建議

---

### 11.3 Phase 1 的 UI 承載方式

UI 是**承載 AI 體驗的容器**，不是主角。Phase 1 需要的 UI：

| UI 元件 | 用途 | 複雜度 |
|---------|------|--------|
| Morning Brief 卡片 | 承載 AI 每日敘事 | 低（靜態 mock 文字）|
| Cmd+K + Quick Capture | 所有捕捉的入口 | 中（送出觸發 mock triage）|
| AI Triage 卡片 | 承載 AI 分類結果 | 中（每筆不同格式）|
| AI Pulse 區塊 | 承載專案 AI 摘要 | 低（每個專案一份 mock）|
| 客戶更新草稿 Dialog | 承載 AI 生成文字 | 低（可編輯 textarea）|
| AI 關係觀察條目 | 承載 AI 關係建議 | 低（mock 列表）|
| 引薦 DM 生成器 | 承載 AI 生成的引薦文字 | 低（可編輯 textarea）|
| 記憶 AI 摘要建議 | 承載 AI 記憶描述 | 低（可確認/修改）|

## 12. 開發方向總結

### 12.1 改寫後的開發方向

**一句話產品定位：**
> Personal OS 是一個以 AI 為核心的個人脈絡作業系統，它主動整理你工作、研究與生活的碎片資訊，幫你每天用最少的認知資源知道「現在最重要的事是什麼、接下來應該做什麼」。

---

**AI 體驗原則：**

1. AI 先做，你確認——不是你做，AI 只是幫你存
2. 敘事而非數字——摘要是故事，不是統計
3. 跨脈絡連結——AI 看見不同模組之間的關係，你不需要手動串
4. 溫和不強迫——提醒是照顧，不是催促
5. 透明有依據——AI 說每一件事都解釋為什麼

---

**核心使用者旅程：**

```
每天：
  打開 → Morning Brief → 確認 2–3 件事 → 關掉
  （10–15 分鐘，不需要打開多個模組）

隨時：
  有想法/訊息/照片 → Cmd+K 丟進去 → AI 整理 → 之後確認

每週：
  看 AI 週摘要（關係、研究成熟度、公司節奏）→ 做幾個決定

特定時機：
  要給客戶更新 → 進專案 → AI 草稿 → 修改 → 複製
  要引薦朋友 → 輸入兩人 → AI DM → 複製
  有研究想法 → Triage 確認歸入哪個主軸
```

---

**AI Agents / AI Roles（概念層，Phase 1 以 mock 模擬）：**

| AI 角色 | 主要功能 | 觸發時機 |
|---------|---------|---------|
| Intake Advisor | 分類、提取、建議歸屬 | 每次 Capture 送出 |
| Morning Narrator | 每日脈絡敘事摘要 | 每天開機 |
| Project Companion | 專案狀態摘要、客戶更新草稿 | 進入專案頁 |
| Research Cartographer | 研究成熟度、想法歸屬、論文關聯 | 新增研究材料或想法 |
| Relationship Observer | 關係動態、引薦時機、DM 草稿 | 每週週期 + 手動觸發 |
| Rhythm Keeper | 公司節奏摘要、財務信號 | 每週 + 財務捕捉 |
| Life Keeper | 溫和生活照護提醒、記憶摘要 | 後台監測 + 記憶捕捉 |

---

**主要資料流：**
```
Capture → [Intake Advisor] → Triage Card
                                    ↓
              [使用者確認] → Entity / Context 更新
                                    ↓
                           [各 AI Agent 分析]
                                    ↓
                            Insight + Suggestion
                                    ↓
                           [使用者決策] → Decision
                                    ↓
                     Action / Memory / PublicOutput
```

---

**Phase 1 最小可行範圍（5 個 AI 情境 + 必要 UI 容器）：**

| 優先 | 內容 |
|------|------|
| P0 | App Shell + Cmd+K Quick Capture |
| P0 | AI Triage Inbox（最核心的 AI 體驗）|
| P0 | Morning Brief 首頁（AI 敘事，非 Dashboard）|
| P1 | AI Project Pulse + 客戶更新草稿 |
| P1 | AI 引薦 DM 生成器 |
| P2 | AI 研究想法歸屬建議 |
| P2 | AI 記憶摘要建議 |
| P2 | AI 關係動態觀察 |
| P3 | AI 週摘要（公司節奏 + 研究成熟度）|

---

**Phase 1 不做什麼：**

| 原計劃項目 | 延後理由 |
|-----------|---------|
| 完整 TanStack Table 資料表（所有模組）| 不是 AI 體驗核心，是資料管理後台 |
| Settings 頁面的完整欄位（LINE/R2/Stream 等）| 後端整合時才有意義 |
| 財務圖表（BarChart/PieChart/LineChart）| 財務信號應出現在摘要中，不需要獨立圖表頁 |
| 研究者資料庫的完整 CRUD | 延後，先建立「AI 建議研究者連結」的流程 |
| 生活事項看板的完整 CRUD | 生活模組入口改為 AI 照護語氣，看板降格 |
| 客戶公開頁面的完整 Tab 設計 | 先做 AI 客戶更新草稿，公開頁後端整合時再做 |
| 多種 Dialog / Sheet 的所有欄位 | 大多數 CRUD 操作延後到後端整合階段 |

---

**原本 UI-heavy 計劃的處理方式：**

| 項目 | 處理方式 | 理由 |
|------|---------|------|
| 工作專案列表 | **保留但簡化**：只要卡片列表，AI Pulse 是主視覺 | 使用者需要能進入專案 |
| 任務列表 | **保留但降格**：作為 AI Pulse 下方的「原始資料」展開 | AI 摘要是主視覺，任務是細節 |
| 商會聯絡人列表 | **保留但簡化**：入口改為 AI 關係觀察，聯絡人列表是次要 | 需要能找到人 |
| 研究主軸卡片 | **保留**：是 AI Research Cartographer 的容器 | 研究是重要的 Entity |
| Settings 頁面 | **大幅簡化延後**：只做 Profile + 語言設定，其他到後端整合 | Phase 1 不需要真實設定 |
| 財務紀錄表格 | **保留但降格**：出現在公司節奏頁的次要區塊 | 使用者確認財務後需要看到列表 |
| 回憶牆 | **保留**：AI Memory Capture 的展示容器 | 記憶體驗需要視覺化容器 |
| 認證頁面 | **完整保留**：UI 品質影響第一印象 | 入口體驗重要 |

### 12.2 Phase 1 三週調整後的週程

**Week 1：AI 核心體驗層**
- App Shell（Sidebar 簡化版 + Header + Cmd+K）
- Morning Brief 首頁（AI 敘事 mock + 動態選擇邏輯）
- Quick Capture → AI Triage Inbox（最核心，要把 AI 卡片體驗做好）
- Mock 資料層 + Entity types

**Week 2：AI 工作 + 研究 + 關係**
- 工作專案列表（簡化卡片）+ AI Project Pulse
- 客戶更新草稿 Dialog（可編輯 + 複製）
- 研究主軸總覽 + AI 研究想法歸屬（Triage 中觸發）
- 商會聯絡人列表 + AI 引薦 DM 生成器

**Week 3：AI 生活 + 公司節奏 + 收尾**
- 生活模組（AI 照護語氣入口 + 事項列表 + AI 記憶摘要）
- 回憶牆 + AI Memory Capture 體驗
- 公司節奏頁（AI 週摘要 + 財務列表）
- 認證頁（Login / Signup）
- 整體收尾：Dark mode、響應式、Toast 通知統一
