## 1. 專案立論：為什麼要做這個系統

### 1.1 問題現象

身為一個同時扮演創業者、研究者、商會成員與生活照顧者的人，每天會在多個身份與情境之間頻繁切換：

* 早上和客戶開會討論交付物
* 中午想到一個研究上的點子
* 下午商會夥伴傳訊息詢問引薦
* 晚上發現感冒症狀又出現、書包該修了

這些事件之間沒有單一的容器，於是它們散落在 LINE、筆記 App、Email、行事曆、腦中。每一次切換都需要重建心智脈絡。

### 1.2 研究依據：這不只是「不夠勤勞」

學術研究指出多重身份切換的真實成本：

* **加州大學研究：頻繁的情境切換可使生產力下降高達 40%。** 知識工作者平均每 3 分鐘切換一次任務，每 10 分鐘切換一次情境，造成持續的轉換成本。
* **微軟研究：典型知識工作者在數位螢幕上停留少於 3 分鐘就會切換到別的事。** 這不是真正的多工，而是注意力的連續被竊取。
* **McKinsey：每週工作時間中，近 20% 消失在尋找內部資訊與追蹤同事上。** 一個良好的知識管理系統可帶來 20–25% 的生產力提升。
* **認知負荷理論（Sweller）：** 工作記憶在處理過多資訊時會超載。沒有外部系統承接，會導致心理混亂、洞察被遺忘、重複勞動。
* **邊界管理理論（Kossek）：** 個人在「整合者（Integrator）」與「分離者（Segmentor）」之間有不同風格，但缺乏「邊界控制感」會直接影響幸福感、工作滿意度與情緒耗竭程度。

### 1.3 本系統要解決的核心問題

**降低「角色切換」的認知摩擦力，同時保留「角色彼此連結」的可能。**

不是追求把工作、研究、生活完全分開（那會切斷研究與工作的洞察轉化），也不是讓所有事情都混在一起（那會造成持續干擾）。系統要做的是：

* 提供每個角色一個清晰的「家」
* 讓使用者主動決定何時跨界、何時隔離
* 在切換時，能快速「載入」該角色目前的脈絡

### 1.4 系統的三條主軸目標

1. **看見全貌：** 無論任何時刻，使用者都能在 60 秒內回答「我手上有哪些事正在發生？哪些需要關注？」
2. **保留脈絡：** 重要的想法、紀錄、關係洞察、回憶，不會因為切換情境而流失
3. **支持轉化：** 學術想法可以轉化為工作洞察，工作經驗可以反哺研究，關係可以連結到合作機會，生活照顧不會因為忙碌就被忽略

## 2. 使用者輪廓與身份切換的負擔

### 2.1 主要使用者：本人（Owner）

一個身兼多重角色的個體：

| 角色                | 主要負擔                                                 |
| ------------------- | -------------------------------------------------------- |
| 創業者 / 公司經營者 | 多個專案並行、客戶溝通、內外帳分流、公司主軸維護         |
| 學術研究者          | 多研究主軸並進、文獻追蹤、發表規劃、研究與工作的雙向轉化 |
| 商會成員            | 人脈經營、引薦撮合、商機追蹤、夥伴協作                   |
| 生活中的人          | 身體照顧、習慣維持、回憶保存、與重要他人的連結           |

### 2.2 次要使用者

* **客戶（Client）：** 不需登入，透過分享連結讀取特定專案的公開內容
* **商會夥伴（Partner）：** 登入後可協作被授權的人脈、商機、引薦範本

### 2.3 「身份切換情境」清單

這些是系統需要正面回應的真實場景：

1. 開完客戶會議，想到研究上的新角度，但 5 分鐘後又有下一個會議
2. 在咖啡廳偶遇一個可以引薦給商會夥伴的人，當下需要快速記錄並啟動後續
3. 半夜想到公司主軸該調整的方向，不想開電腦但又怕忘記
4. 客戶問「上次提的那個交付物進度如何」，需要立即取出對外可見的資訊
5. 收到研討會 CFP，想知道哪個研究主軸最適合投，截止日期是否與工作衝突
6. 連續加班三週後發現感冒沒看完、書包還沒修，需要被提醒卻又不想被罪惡感淹沒

## 3. 核心設計哲學

### 3.1 雙視角原則：內部 / 對外

工作與商會兩個模組設計上必須區分：

* **內部視角（Internal）：** 我自己思考、追蹤、規劃用的內容。包含工作中的內部討論、關係洞察、未公開的判斷。
* **對外視角（External）：** 客戶 / 夥伴可以看到的內容。聚焦於交付、進度、共識資料。

切換是顯性動作。任何欄位、任何文件，使用者都應該清楚知道目前在哪一個視角。

**為什麼這樣設計：** 避免心理上的「自我審查」污染原本應該是私密思考的空間。研究顯示，當邊界感不清晰時，人會傾向於「保守化」自己的紀錄，導致最關鍵的洞察被磨平。

### 3.2 捕捉優先於組織

PKM 領域的常見失敗模式：花三個週末打造完美的 Notion Dashboard，最後沒有任何內容被填進去。

本系統的設計原則：

* **捕捉摩擦力極低：** 任何模組都支援「快速新增 → 之後再分類」
* **組織延後處理：** 預設都會落入該模組的 Inbox，使用者可選擇何時整理
* **每週 Review 而非每天微調：** 系統不期待使用者每天精修分類

### 3.3 表達就是知識的活化

知識管理框架（如 Tiago Forte 的 CODE：Capture → Organize → Distill → Express）強調，沒有 Express 的知識管理就是囤積。本系統因此特別重視「轉化輸出」這件事：

* 研究主軸有獨立的「轉化輸出」欄位（簡報、海報、學習路徑、摘要）
* 工作專案有獨立的「對客戶交付物」欄位
* 公司主軸鼓勵被引用到專案描述中

### 3.4 不把生活變成績效

生活模組不採用標準的 To-Do List 形式。原因：把身體照顧、回憶、人際情感變成待辦清單，會引入績效焦慮，最終讓使用者迴避這個模組。

設計上：

* 用「下次該做」而非「該做卻沒做」的語氣
* 完成的不是「打勾」，而是「記錄一次發生」
* 回憶模組強調保存而非提醒

## 4. 模組一：工作管理 — 內外雙視角的專案中樞

### 4.1 觸發情境

**情境 A：早上 9:30 — 客戶來電**

> 客戶 Lisa 來電：「上次討論的 Q2 數據儀表板，我們今天能不能對一下進度？」使用者需要在 30 秒內：
>
> * 找到這個專案
> * 看到目前對 Lisa 公開的最新狀態
> * 想起內部上次討論的卡點是什麼

**情境 B：晚上 11 點 — 想到一個專案的解法**

> 洗澡時突然想到某個客戶專案的技術解法。怕忘記，但不想花 10 分鐘整理思路。需要：
>
> * 30 秒內把想法丟進對的位置
> * 之後能找到、且知道屬於哪個專案

**情境 C：商會聚會中 — 客戶詢問**

> 商會午餐時，客戶當場問「我看一下我們專案目前有什麼資料可以給我」。使用者在手機上：
>
> * 打開該專案的對外連結
> * 直接傳送給客戶查看（不必額外整理）

### 4.2 系統如何回應

1. **專案有兩個分頁：「我的工作區」與「客戶可見區」。** 兩個區是同一個專案的不同面向，預設新增的內容都進「我的工作區」，需要手動「公開」才會出現在客戶可見區。
2. **每個專案有一個唯一的「客戶連結」。** 點開即可看到對外公開的進度、交付物、參考資料。連結可隨時撤銷或重新產生。
3. **快速捕捉入口：** 在任何頁面，都可用快捷鍵（或手機底部按鈕）「丟想法到專案」，預設目的地是最近開啟的專案。

### 4.3 此體驗解決的議題

| 議題                                   | 解決方式                                                  |
| -------------------------------------- | --------------------------------------------------------- |
| 內部思考會被「客戶看得到嗎？」自我審查 | 預設私密、明確標記公開動作                                |
| 客戶頻繁來電打斷工作節奏               | 客戶連結 = 對外資訊的單一來源；客戶自助查詢，減少臨時詢問 |
| 想法在「值得不值得記」的猶豫中流失     | 30 秒內可捕捉、之後再決定要不要保留                       |
| 多客戶平行進行時搞不清楚誰是誰         | 每個專案有清晰的客戶名稱、最後互動時間、下一步動作        |

### 4.4 對使用者的影響

**短期（前 2 週）：** 從「混亂感」變成「至少知道有什麼正在進行」。即使內容還沒整理好，光是有清單可看就會降低焦慮。

**中期（1–3 個月）：** 開始建立客戶溝通的節奏。客戶連結讓「我有準時更新」這件事變得可見，建立信任。

**長期（半年以上）：** 累積「專案歷史」資產。新進客戶、新合作夥伴、未來的自己，都能從過去的專案紀錄中快速理解過往做過什麼、產出什麼、學到什麼。

## 5. 模組二：商會 CRM — 關係資產與引薦工程

### 5.1 觸發情境

**情境 A：午餐遇到一個有趣的人**

> 在商會午餐遇到 David，他在做東南亞物流。閒聊中發現他可能可以幫到夥伴 Allen 的進口生意。當下使用者需要：
>
> * 記下 David 是誰
> * 標註他與 Allen 之間的潛在連結
> * 之後跟進是否真的能促成

**情境 B：商會夥伴需要被引薦**

> 夥伴 Allen 傳訊：「上次說可以幫我介紹做物流的，可以動了嗎？」使用者需要：
>
> * 找出之前記錄的潛在連結
> * 拿出引薦範本（DM）
> * 確認雙方的時間與意願

**情境 C：商會內部協作**

> 三個商會夥伴一起經營一條人脈線，需要共同維護同一份聯絡資訊與引薦進度。

### 5.2 系統如何回應

1. **聯絡人卡片：** 每個人有姓名、公司、來源、見面情境、私人洞察（僅自己可見）、共享筆記（夥伴可見）。
2. **關係圖：** 可以標記「A 和 B 之間可能有合作機會 / 已成案 / 已嘗試但失敗」，附上「我在這段關係中扮演的角色」。
3. **商機看板：** 漏斗階段（探索中 → 洽談中 → 成交 / 失敗）。可以連結到具體聯絡人。
4. **DM 引薦範本庫：** 多種情境的範本（推薦給某類專業 / 邀請見面 / 跟進進度），夥伴可以共用編輯。
5. **共享開關：** 每筆聯絡人、每個商機都可以選擇「私人」或「商會夥伴可見」。

### 5.3 此體驗解決的議題

研究指出，**一個透過熱介紹（warm introduction）認識的決策者，回應率是冷觸達的 15 倍。** 但熱介紹仰賴「我知道我認識誰」與「我記得他們的脈絡」。這正是使用者目前最容易忘記的事。

| 議題                                           | 解決方式                             |
| ---------------------------------------------- | ------------------------------------ |
| 認識很多人，但忘記誰可以引薦給誰               | 關係圖讓潛在連結被記下，未來可被觸發 |
| 私人的關係洞察與夥伴共用資訊混在一起，怕說錯話 | 雙層權限：私人筆記 vs 共享筆記       |
| 引薦時每次都要重新寫文字，文字品質不一         | 範本庫，夥伴共同優化                 |
| 商機進度散在 LINE、Email、腦中                 | 商機看板 = 商會合作的單一儀表板      |

### 5.4 對使用者的影響

**對個人：** 從「靠記憶力」變成「靠系統」。讓使用者敢於認識更多人，因為知道不會在認識下一位時忘記前面那位。

**對商會：** 形成集體的關係資產。夥伴離開或加入時，關係知識不會流失。共享範本帶來引薦品質的一致性。

**長期：** 個人成為商會中的「樞紐節點」。當系統清楚顯示誰連結誰，使用者扮演引薦者的能力會直接強化。

## 6. 模組三：公司管理 — 主軸與節奏的存放處

### 6.1 觸發情境

**情境 A：又被問到「你們公司在做什麼」**

> 投資人問、潛在客戶問、新員工問。使用者每次回答都不太一樣，有時甚至前後矛盾。需要一個「定版」可以參考。

**情境 B：發現某個專案偏離公司主軸**

> 接了一個案子做了一半，發現它和公司核心其實沒關係。使用者需要回頭檢視當時為什麼接這個案子、是否要繼續。

**情境 C：公司方向需要調整**

> 經營一年後，使用者意識到當初的策略需要修正。需要記下：原先方向、修正後方向、為什麼修正、什麼時候做這個決定。

### 6.2 系統如何回應

1. **公司核心文件區：** 願景、策略、節奏、當前推進方向，各自獨立的 Markdown 文件。
2. **置頂機制：** 最重要的 1–3 份文件可以置頂，每次進入公司模組就先看到。
3. **版本歷史：** 重要文件的修訂保留歷史，可以看到「什麼時候改的、改了什麼」。
4. **與專案的連結：** 工作模組的專案可以標註「對應公司主軸：XXX」，反向也能在公司主軸頁面看到「目前有哪些專案對應這條主軸」。

### 6.3 此體驗解決的議題

| 議題                                           | 解決方式                               |
| ---------------------------------------------- | -------------------------------------- |
| 公司方向感隨工作壓力漂移                       | 文件作為錨點，每週 Review 時可對照     |
| 策略修正缺乏紀錄，幾年後忘了當初為什麼這樣決定 | 版本歷史保留決策脈絡                   |
| 接案標準模糊，常接到不該接的                   | 主軸文件成為「該接哪些案子」的判斷依據 |

### 6.4 對使用者的影響

**穩定性：** 公司不再是「腦中的一個模糊願景」，而是可以被引用、檢視、修正的具體文件。

**一致性：** 對外溝通公司價值時有可靠的版本可依據，不會每次說的都不一樣。

**反思性：** 過去的策略決定變成可學習的素材。未來犯類似錯誤時，能看到「上次怎麼判斷的」。

## 7. 模組四：財務管理 — 內外帳分流與專案歸屬

### 7.1 觸發情境

**情境 A：客戶付款入帳**

> 客戶 Lisa 匯了第二期款。使用者需要記錄：金額、對應專案、外帳（會計記帳用）。同時自己要知道這個專案目前實際淨利多少（內帳）。

**情境 B：報稅季節**

> 會計事務所要外帳資料。使用者需要快速匯出該年度所有外帳紀錄。

**情境 C：判斷是否該接下一個案子**

> 一個新案子報價猶豫。使用者想看「過去類似規模的案子，扣掉真實成本後，淨利是多少」。

### 7.2 系統如何回應

1. **每筆紀錄有兩個帳本標記：內帳 / 外帳。** 同一筆收入可能只進外帳（正式發票），但也有些只進內帳（內部分潤、夥伴分成等）。
2. **每筆紀錄可關聯到專案。** 讓「這個專案到底賺了多少」變成可查詢的資訊。
3. **月度 / 季度 / 年度摘要：** 收支趨勢、分類佔比、各專案毛利。
4. **CSV 匯出：** 給會計、給自己看歷史。

### 7.3 此體驗解決的議題

| 議題                                 | 解決方式                                   |
| ------------------------------------ | ------------------------------------------ |
| 內帳外帳混在一起，報稅時要花一週整理 | 標記分流，匯出即用                         |
| 接案報價憑感覺，實際做完才發現虧損   | 專案歷史毛利可查                           |
| 公司現金流不透明，焦慮但不知從何看起 | 摘要圖表提供「我這個月過得如何」的快速答案 |

### 7.4 對使用者的影響

**短期：** 從「不敢看財務」到「知道財務狀況」。即使數字不漂亮，知道實情比逃避有助益。

**中期：** 報價能力提升。透過歷史毛利資料，可以更精準預估新案子。

**長期：** 公司成為一個可以被分析、被優化的對象，而非一個靠靈感經營的混沌體。

## 8. 模組五：學術研究 — 想法、人脈、節奏的三軸並行

### 8.1 觸發情境

**情境 A：搭捷運時想到一個研究問題**

> 「如果把工作上做的這個流程，套上 XX 理論框架，會不會就是一個有趣的研究？」使用者需要：
>
> * 立刻記下這個連結
> * 標註它可能屬於哪個研究主軸
> * 標註它與哪個工作專案有關

**情境 B：讀到一篇相關論文**

> Google Scholar 跳出一篇看起來相關的論文，作者是個沒見過的研究者。使用者需要：
>
> * 把論文記入該研究主軸的材料庫
> * 把作者加入「相關研究者」資料庫
> * 之後追蹤這個研究者還發了什麼

**情境 C：研討會 CFP 出現**

> 某研討會的徵稿截止剩 6 週。使用者需要判斷：
>
> * 哪個研究主軸最適合投這個 venue
> * 6 週內可以產出什麼程度的稿件
> * 期間有沒有工作上的衝突

**情境 D：把研究轉化成可以給別人看的東西**

> 學期結束要做一份研究海報。使用者需要從累積的想法、材料、結論中抽取重點，做成一個對外的「展示版本」。

### 8.2 系統如何回應

1. **研究主軸是頂層單位。** 每個主軸是一個獨立空間，包含：
   * **想法（Ideas）：** 概念、假設、問題、可轉化到工作的點
   * **材料（Materials）：** 論文、書籍、資料、工具
   * **相關研究者（Researchers）：** 作者資料庫，連結到他們的論文
   * **發表目標（Publication Targets）：** 適合的研討會/期刊/平台與截止日
   * **里程碑（Milestones）：** 進度節點與交付項目
   * **轉化輸出（Outputs）：** 簡報、海報、摘要、學習路徑、部落格
2. **想法可標記「工作連結」：** 一條想法可以同時屬於研究主軸 A，也標記「可能適合應用於專案 X」。這建立研究與工作的雙向通道。
3. **研究者追蹤：** 看到一篇感興趣的論文，3 步加入研究者資料庫並標記與哪條主軸相關。
4. **發表行事曆視圖：** 把所有發表目標的截止日疊加成行事曆，看到 6 個月內的密度。

### 8.3 此體驗解決的議題

| 議題                                               | 解決方式                             |
| -------------------------------------------------- | ------------------------------------ |
| 研究想法散在便利貼、Notion、Google Doc，找不到歷史 | 集中在主軸下，且永遠可被搜尋         |
| 學術與工作完全切開，洞察無法雙向流動               | 想法可標記工作連結，建立轉化通道     |
| 不知道現在世界上誰在做類似研究                     | 研究者資料庫累積成個人的「領域地圖」 |
| 研究進度感很差，經常感覺什麼都沒推進               | 里程碑 + 輸出讓進展可見              |
| CFP 截止日散在 Email，忘記投稿                     | 發表行事曆顯示 6 個月內的截止熱度    |

### 8.4 對使用者的影響

**身份完整性：** 使用者不必選擇「我是研究者還是工作者」。系統承認兩者並存，並讓兩者互相滋養。

**生產節奏：** 從「靈感來了才做研究」變成「研究是有節奏的」。不必每天大步前進，但有里程碑提醒、有輸出累積。

**領域定位：** 透過追蹤相關研究者，使用者逐漸看清楚自己在領域中的位置 — 誰是夥伴、誰是先行者、自己的獨特角度在哪裡。

## 9. 模組六：生活管理 — 維護自己與保存記憶

### 9.1 觸發情境

**情境 A：感冒第三次復發**

> 又感冒了。回想起來，似乎這幾個月都有類似症狀，但每次都覺得「等下就會好」。需要：
>
> * 看到這是第幾次
> * 提醒該去看醫生不是「之後」而是「下週」

**情境 B：朋友生日要送禮物**

> 朋友 Mark 下個月生日。使用者想做一個有心意的禮物，不是隨便買。需要：
>
> * 翻出和 Mark 過去的相處片段
> * 從中找出一個值得紀念的瞬間
> * 規劃送禮形式

**情境 C：書包壞了三個月**

> 書包的拉鍊壞了三個月，每次提包都不順手，但忙碌之中始終忘記去修。需要：
>
> * 一個不會讓使用者罪惡感爆棚的提醒方式
> * 不被工作待辦淹沒

**情境 D：半年後想回看「今年我經歷了什麼」**

> 年底想回顧。發現大部分回憶模糊，只剩下幾個重大事件。需要：
>
> * 一個累積的「人生片段」檔案

### 9.2 系統如何回應

1. **生活事項看板（Life Board）：** 分類顯示：健康、身體照顧、習慣、雜務、感情關係。
2. **「下次該做」而非「未完成」：** 同樣是修書包，文字呈現是「修書包 — 下次建議：本週」而非「逾期 90 天」。
3. **週期性事項自動更新：** 完成「定期看醫生」後，下次提醒日期自動依設定週期推進。
4. **回憶牆（Memory Wall）：**
   * 可記錄：標題、敘事、發生日期、相關的人
   * 支援照片附件
   * 可置頂重要回憶
   * 可依「人」或「時間」篩選
5. **與日常切割：** 生活模組獨立於工作 / 研究的待辦清單。完成度不會出現在工作儀表板，避免混雜。

### 9.3 此體驗解決的議題

| 議題                                     | 解決方式                               |
| ---------------------------------------- | -------------------------------------- |
| 身體警訊被工作壓過去，最後變成大問題     | 健康事項有獨立空間，不被工作待辦稀釋   |
| 待辦清單模式讓生活變成績效，最終逃避不看 | 用「下次該做」溫和語氣，沒有過期的紅字 |
| 想送有心意的禮物，但記不起共同回憶       | 回憶牆可依「相關的人」篩選             |
| 生命片段流失，年底回顧空空如也           | 持續累積，不必每天紀錄                 |

### 9.4 對使用者的影響

**自我關懷：** 系統明確告訴使用者「這些事很重要、值得有自己的空間」。這本身是一種訊息。

**情感資產：** 與重要他人之間的互動不再仰賴記憶力。多年後，仍可從回憶牆中看到關係的軌跡。

**長線健康：** 反覆出現的健康訊號（如反覆感冒、反覆腸胃不適）會被系統視覺化，成為與醫生溝通的有效依據。

## 10. 跨模組整合：每日入口與身份切換體驗

### 10.1 每日 Dashboard

打開系統後第一個畫面，回答三個問題：

1. **今天有什麼必須處理的？** 工作專案的今日任務 + 研究里程碑的截止 + 生活事項的高優先提醒
2. **這週有什麼值得關注的？** 商會引薦進度、發表目標倒數、財務異常
3. **我今天想思考什麼？** 顯示最近捕捉但尚未整理的想法，鼓勵 5 分鐘整理

### 10.2 身份切換（Mode Switch）

側邊欄提供六個模組入口，但更重要的是：每個模組打開時，系統會用視覺語言（顏色、字型、留白）暗示「現在在哪個身份」。

* 工作模組：節制、結構化、面向交付
* 研究模組：寬鬆、可探索、面向想法
* 商會模組：人物中心、關係視覺化
* 生活模組：柔和、低壓、回憶優先

**為什麼要這樣設計：** 邊界管理理論指出，視覺/環境提示可以幫助大腦在心理上完成「角色轉換」。一致的 UI 反而會讓多角色使用者感覺所有事情混在一起。

### 10.3 跨模組連結

雖然模組分離，但下列連結是核心：

* 研究想法 ↔ 工作專案（研究啟發工作、工作素材回饋研究）
* 工作專案 ↔ 公司主軸（這個案子對應哪條策略）
* 工作專案 ↔ 財務紀錄（這個案子實際賺多少）
* 商會聯絡人 ↔ 工作專案（這個客戶是怎麼來的）
* 回憶 ↔ 商會聯絡人 / 朋友（與這個人有過什麼）

### 10.4 快速捕捉（Quick Capture）

在任何頁面（包含手機）都可呼叫一個小視窗：

* 一行文字輸入
* 自動偵測或手動指定目的地（工作、研究、生活、商會、想法暫存區）
* 30 秒內回到原本工作

這是降低「捕捉摩擦力」的核心機制。

## 11. 技術架構選擇與理由

### 11.1 技術棧

| 層            | 技術                     | 理由                                                          |
| ------------- | ------------------------ | ------------------------------------------------------------- |
| 前端框架      | Next.js 14 (App Router)  | Server Components 適合資料密集儀表板；Route Handlers 簡化 API |
| 樣式          | Tailwind CSS + shadcn/ui | 高速建構但保留設計彈性；模組可有不同視覺語言                  |
| 後端 / 資料庫 | Supabase (PostgreSQL)    | RLS 直接在資料庫層保證權限；少寫後端                          |
| 認證          | Supabase Auth            | 支援 Magic Link、OAuth；客戶連結使用 token-based 機制         |
| 儲存          | Supabase Storage         | 附件、回憶照片、研究材料 PDF                                  |
| 即時          | Supabase Realtime        | 商會夥伴協作的同步更新                                        |
| 部署          | Vercel + Supabase Cloud  | 兩者皆有免費額度，適合個人系統起步                            |

### 11.2 路由架構（Next.js App Router）

```
app/
├── (auth)/              # 登入、註冊
├── (dashboard)/         # 主應用，需登入
│   ├── page.tsx         # 每日 Dashboard
│   ├── work/            # 工作模組
│   ├── chamber/         # 商會 CRM
│   ├── company/         # 公司管理
│   ├── finance/         # 財務管理
│   ├── research/        # 學術研究
│   └── life/            # 生活管理
├── client/[token]/      # 客戶公開頁面（無需登入）
└── api/                 # Route Handlers
```

### 11.3 權限模型

三層使用者：

* **Owner：** 系統擁有者，全部讀寫
* **Partner：** 商會夥伴，僅能讀寫被授權為「共享」的商會資料
* **Client：** 客戶，透過分享 token 讀取被標記為「對外公開」的工作專案內容

權限透過 Supabase Row Level Security (RLS) 在資料庫層執行，避免應用層權限漏洞。

### 11.4 客戶連結機制

* 每個工作專案有一個唯一 token（隨機產生）
* 客戶透過 `/client/[token]` 訪問，無需註冊
* 訪問時 server-side 驗證 token 有效性
* 只能看到專案中標記為「客戶可見」的任務與交付物
* token 可隨時撤銷或重新產生

### 11.5 快速捕捉的技術實作

* 全域 Cmd+K（手機為底部 FAB 按鈕）開啟模態
* 文字輸入後送至 `/api/capture` 端點
* 後端自動依關鍵詞 / 模組標籤分流到對應 Inbox
* 使用者可日後在各模組的 Inbox 區整理

## 12. 非功能性需求

### 12.1 可用性

* **首次載入 < 2 秒（Dashboard）：** 透過 Server Components + 串流避免阻塞
* **快速捕捉 < 500ms：** 從快捷鍵到輸入框出現
* **手機優先：** 大量場景在手機上發生（搭車、開會、商會聚會），所有核心操作必須在 390px 寬度上順暢運作

### 12.2 安全性

* 所有資料表強制啟用 RLS
* 客戶連結 token 使用 HMAC 簽章，可選配有效期
* 客戶連結支援撤銷
* 環境變數不進版本控制

### 12.3 隱私

* 個人筆記與共享筆記在資料庫層分離
* 商會夥伴退出時，私人欄位資料完整保留給 Owner
* 系統不對使用者輸入做任何分析（不訓練模型、不行為追蹤）

### 12.4 韌性

* 重要資料（專案、研究、回憶）每日自動備份到 Supabase Storage
* 提供 JSON 全資料匯出（避免使用者被鎖死）
* 客戶連結與分享資料即使主帳號暫停仍可被重新產生

### 12.5 可擴展性

* 模組之間透過明確 API 邊界溝通，未來增加第七、第八模組（如：閱讀清單、健身紀錄）不會大改架構
* 支援未來引入 AI Assist（如：自動分類捕捉的想法、提示研究者新發表）

## 13. 開發階段規劃

### Phase 0 — 基礎建設（1 週）

* Supabase 專案、Auth、儲存
* Next.js 路由架構、共用 UI 元件
* Dashboard 框架（先空殼）
* 快速捕捉機制（Inbox-only 版本）

### Phase 1 — 工作管理（2 週）

> 優先做這個的原因：使用者每天最先打開的是工作模組，先把這部分做出來能立刻減輕日常負擔。

* 專案 CRUD + 雙視角切換
* 客戶連結機制
* 任務、紀錄、交付物管理

### Phase 2 — 學術研究（2 週）

> 第二優先：研究是時間敏感的（截止日、轉化）。早做能立刻接住目前在發生的研究想法。

* 研究主軸 + 想法 + 材料
* 研究者資料庫
* 發表目標 + 里程碑
* 轉化輸出區

### Phase 3 — 商會 CRM（1.5 週）

* 聯絡人 + 關係圖
* 商機看板
* DM 範本（含夥伴共用）
* 夥伴邀請與權限

### Phase 4 — 生活管理（1 週）

> 雖然優先級看似較低，但越早做越能避免「自我消耗」。

* 生活事項看板（含週期性）
* 回憶牆

### Phase 5 — 公司 + 財務（1.5 週）

* 公司核心文件 + 版本歷史
* 財務紀錄 + 內外帳分流
* 摘要圖表 + CSV 匯出

### Phase 6 — 整合層（1 週）

* 每日 Dashboard 整合所有模組
* 跨模組連結（研究 ↔ 工作、聯絡人 ↔ 專案 等）
* 模組間視覺語言差異化

### Phase 7 — 優化與韌性（1 週）

* 全資料匯出
* RLS 完整審查
* 手機體驗最終調整
* 效能優化

**預估總期程：** 11 週（約 2.5 個月）

## 結語

這份系統的本質不是「另一個任務管理工具」，而是一個 **幫助多重身份個體，在不放棄任何角色的前提下，仍能保有清晰、保有從容、保有自我關懷的結構化容器** 。

每一個模組的存在，都是因為某個生活情境正在反覆耗損使用者的能量。系統的成功標準，不是被打開的次數，而是：

* 在客戶突然來電時，使用者可以從容地說「我看一下」並 30 秒內回應
* 在搭捷運時想到的研究點子，三個月後仍能被找回並發展
* 在年底回顧時，發現自己這一年的人生片段都被好好保存著
* 在身體不適時，意識到「這已經不是第一次了」並啟動處理

**降低切換的負擔，但不切斷彼此的滋養。**這是 Personal OS 的承諾。
