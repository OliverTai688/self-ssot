## Current Operating Rule

BFF-first development:

```txt
UI need
  -> BFF contract
  -> server action / route handler / Server Component loader
  -> requireUser()
  -> service-layer authorization
  -> domain service
  -> Prisma or approved adapter
  -> mapper / view model
  -> Client Component interaction
```

For UI-only/mock tasks: stop before server actions, database writes, connector runtime, or module SSOT writes.

For non-trivial issues: run the Research-To-Task Quality Gate before implementation. Review local docs/code, use official docs or reference website/product implementation patterns when relevant, then convert findings into executable scope, acceptance criteria, likely files, verification, and risks.

For page-level UI/settings/admin/module/frontstage/workflow tasks: score page requirement understanding from 0 to 100 before implementation. Low 0-59 requires 5 same-issue research optimization rounds, medium 60-79 requires 4, and high 80-100 requires 3. Each round must use a distinct lens, refine the same page requirement, and record selected/rejected implementation patterns before executable task shape is created.
